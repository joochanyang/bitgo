# 설계서: 성장하는 AI 트레이딩 에이전트 (Bull/Bear/Risk + 장기기억)

작성일: 2026-06-30 · 프로젝트: go-bot (ANTIGRAVITY) · repo `github.com/joochanyang/bitgo`

---

## 1. 목표와 철학

**한 줄 목표**: 4시간 주기로 고정 룰을 반복하는 기계가 아니라, 시장을 지켜보다 "지금이 타이밍"이라 판단되면 진입하고, 보유 중에도 계속 생각하며 대응하며, 거래를 거듭할수록 과거에서 배우는 — 사람 트레이더처럼 사고·성장하는 AI 에이전트.

**운영 철학**:
- **소액 실전 → 검증 → 고액**: 지금은 소액으로 시작하되, 이건 "고액을 믿고 맡길 수 있는가"를 데이터로 증명하는 준비 단계다. 따라서 판단 근거·성과 추적·리스크 관리가 특히 중요하다.
- **검증 없는 실거래 전환 금지**: 새 두뇌는 반드시 페이퍼(모의)에서 성과를 입증한 뒤 실거래로, 실거래에서 다시 입증한 뒤 고액으로 올린다.
- **기존 실거래 룰봇(volatility_breakout)은 건드리지 않는다**: 새 AI 에이전트는 완전히 별개 경로로 페이퍼 검증부터 시작한다. 현재 50 USDT 실거래는 그대로 유지.

**근거 (2026 현황 조사)**: 트레이딩 성패는 "어떤 LLM이냐"보다 **에이전트 아키텍처**가 좌우한다(모델 차이는 결과에 미미). 검증된 우승 패턴 = 다중 에이전트(Bull/Bear/Risk) + 계층형 장기기억(FinMem, TradingAgents). 단일 LLM에 공개 데이터를 묻는 방식은 "factor crowding"으로 이미 무력화됨 → 차별화는 우리만의 기억·관점에서 나온다.

## 2. 아키텍처 개요

```
[이벤트 트리거]  ← 실시간 WS 가격 스트림 + 지표 감시 (싸게 상시 감시)
      │  의미있는 움직임(돌파·급변동·모멘텀 전환·보유포지션 위험) 감지 시에만 발화
      ▼
[컨텍스트 빌더]  ← 현재 시장 스냅샷(가격·지표·포지션) + 장기기억에서 유사 과거사례 회수
      │
      ▼
[다중 에이전트 심의]
   ├─ Bull 에이전트:  "살(유지할) 이유" 주장 + 근거
   ├─ Bear 에이전트:  "사지 말(청산할) 이유" 주장 + 근거
   └─ Risk 감독관:    양측 + 기억 + 리스크규칙 종합 → 최종 결정(JSON)
      │  결정 = {action, size, stopLoss, takeProfit, confidence, reasoning}
      ▼
[리스크 가드]  ← 결정론적 안전장치(코드). AI 결정을 검증·클램프(절대 우회 불가)
      │  포지션캡·리스크%·SL필수·잔고확인·소액계좌 사이징 보정 등
      ▼
[실행기]  ← 페이퍼 or 실거래 (기존 exchange 인터페이스 재사용)
      │
      ▼
[결과 기록]  ← 진입~청산 전 과정·근거·결과(PnL)를 장기기억 DB에 누적
      │
      └──────→ (다음 트리거 때 회수되어 학습에 활용)

[성과 추적기]  ← 누적 거래 통계(승률·기대값·낙폭) → 전환 기준 충족 시 사용자에게 단계 상향 제안
```

**핵심 설계 원칙**: AI는 "판단"만 하고, **돈에 직접 닿는 안전장치는 전부 결정론적 코드(리스크 가드)**가 쥔다. AI가 환각으로 위험한 주문을 내려 해도 가드가 막는다. 이게 AI 트레이딩의 생명줄이다.

## 3. 컴포넌트 명세

각 컴포넌트는 단일 책임 + 명확한 인터페이스 + 독립 테스트 가능하게 설계한다. 기존 `pkg/` 구조와 패턴을 따른다.

### 3.1 `pkg/agent/trigger` — 이벤트 트리거
- **하는 일**: 실시간 가격/지표를 감시하다 "AI를 깨울 가치가 있는" 이벤트를 발화. 평소엔 싼 계산만(AI 호출 0).
- **트리거 조건(초기)**: ① 변동성 돌파 채널 이탈 ② 급변동(N분 X% 이동) ③ 보유 포지션이 SL/TP 근접 ④ 모멘텀 지표 전환. 각 조건은 설정값으로 조정 가능.
- **인터페이스**: `Watch(ctx) <-chan TriggerEvent`. 이벤트엔 종류·심볼·스냅샷 포함.
- **의존**: 기존 `pkg/exchange/pubws.go`(실시간 스트림), `pkg/indicators`.
- **쿨다운**: 같은 심볼 연속 발화 방지(설정값). 비용·과민반응 억제.

### 3.2 `pkg/agent/memory` — 장기기억 (결과 회고 학습)
- **하는 일**: 모든 거래의 {진입시점 상황 스냅샷, 각 에이전트 주장, 최종 결정·근거, 결과(PnL·보유시간·청산사유)}를 저장. 새 판단 시 **현재 상황과 유사한 과거 사례 K건을 회수**해 컨텍스트로 제공.
- **유사도 회수(초기)**: 규칙 기반(심볼·시장국면·지표구간 매칭)으로 시작. 임베딩 벡터검색은 나중에 확장(YAGNI — 초기엔 불필요).
- **저장소**: SQLite 또는 기존 `pkg/db` 패턴의 JSON. 결정은 구현 단계에서(데이터량 적으니 단순 우선).
- **인터페이스**: `Record(TradeEpisode)`, `Recall(situation) []TradeEpisode`, `Stats() PerformanceSummary`.
- **회고**: 청산 시 결과를 해당 에피소드에 채워넣어 "이 판단이 옳았나"를 사후 라벨링.

### 3.3 `pkg/agent/council` — 다중 에이전트 심의
- **하는 일**: Bull·Bear·Risk 세 역할을 LLM 호출로 수행. Bull/Bear는 병렬, Risk가 종합.
- **Bull/Bear**: 같은 컨텍스트(시장+기억)를 받되 상반된 역할 프롬프트. 각자 주장+근거+확신도 반환.
- **Risk 감독관**: 양측 주장 + 기억의 교훈 + 현재 포지션/리스크 상태를 받아 **구조화된 결정(JSON 강제)** 반환: `{action: ENTER_LONG|ENTER_SHORT|HOLD|CLOSE|PARTIAL_CLOSE|ADJUST_SL, sizePct, stopLossPct, takeProfitPct, confidence, reasoning}`.
- **모델 추상화**: `LLMClient` 인터페이스 뒤로 숨김(기존 `pkg/ai` 확장). Claude/Gemini/GPT 교체가 설정 한 줄. 초기 모델은 사용자 선택(미정 — 키 발급 시 확정).
- **인터페이스**: `Deliberate(ctx, Context) (Decision, error)`. 실패·타임아웃 시 안전 기본값(HOLD) 반환.

### 3.4 `pkg/agent/guard` — 리스크 가드 (결정론적, 최우선 안전장치)
- **하는 일**: AI의 `Decision`을 받아 **코드 규칙으로 검증·보정**한 뒤에야 실행 허가. AI는 절대 이걸 우회 못 함.
- **규칙**: ① SL 없으면 진입 거부 ② 포지션당 risk% 상한 클램프 ③ 포트폴리오 합산 리스크/마진 캡 ④ 잔고 조회 실패 시 진입 차단 ⑤ 소액계좌 최소주문수량 보정(감사에서 발견한 BTC 71% 묶임 문제) ⑥ confidence 임계 미만이면 HOLD 강등 ⑦ 비정상값(0·NaN·음수) 거부.
- **인터페이스**: `Validate(Decision, AccountState) (SafeDecision, []Rejection)`. 거부 사유를 기억에도 기록.
- **재사용**: 기존 `pkg/strategy/sizing.go`의 검증 로직을 최대한 재사용.

### 3.5 `pkg/agent/runner` — 오케스트레이터
- **하는 일**: 위 컴포넌트를 엮는 루프. trigger→memory.recall→council→guard→execute→memory.record. 기존 `pkg/exchange.Exchange`(페이퍼/실거래)와 `pkg/notify`(텔레그램) 재사용.
- **격리**: 기존 `pkg/bot/engine.go`(룰봇)와 **완전 별개**. 새 진입점 `cmd/agent` 또는 엔진에 모드 플래그.

### 3.6 성과 추적 + 전환 제안
- **하는 일**: 누적 거래로 승률·기대값(R-multiple)·최대낙폭·표본수 계산. **기준 충족 시 사용자에게 "다음 단계(페이퍼→실거래 / 증액) 제안" 알림**(텔레그램). 전환 실행은 항상 사용자 승인.
- **초기 기준(조정 가능)**: 예) 페이퍼 최소 30거래 & 승률≥X% & 기대값>0 & 최대낙폭<Y% → 실거래 소액 제안. 실거래에서 재충족 → 증액 제안.

## 4. 데이터 흐름 (한 사이클)

1. WS 스트림이 BTC 급등 감지 → trigger가 `TriggerEvent{급변동, BTCUSDT, 스냅샷}` 발화.
2. 컨텍스트 빌더가 현재 지표 + memory.Recall("BTC 급등 유사상황 과거 5건") 결합.
3. Bull("돌파 유효, 거래량 동반 → 진입"), Bear("RSI 과매수, 과거 유사사례 3건 중 2건 손실") 병렬 주장.
4. Risk 감독관: "기억상 이 패턴 승률 낮음 + 과매수 → confidence 0.4, HOLD" 결정.
5. 가드: HOLD는 무위험 → 통과. 실행 없음.
6. memory.Record(이번 심의·결정). 다음에 같은 상황 오면 이 판단도 참고됨.
   *(반대로 ENTER 결정이면: 가드가 size/SL 검증·보정 → 실행기가 페이퍼/실거래 주문 → 텔레그램 알림 → 청산 시 결과를 에피소드에 회고 기록)*

## 5. 에러 처리 · 안전

- **LLM 실패/타임아웃**: 심의 실패 시 항상 안전 기본값 HOLD. 보유 포지션의 기존 SL은 거래소에 이미 걸려 있어 보호 유지.
- **가드 우선**: AI 결정과 무관하게 돈에 닿는 모든 행위는 가드 통과 필수. 가드 자체는 LLM 없는 순수 코드 → 단위 테스트로 강하게 검증.
- **기존 실거래 룰봇 불간섭**: 새 코드는 별 패키지·별 진입점. 룰봇 엔진/설정 미변경.
- **비용 가드**: 트리거 쿨다운 + 일일 AI 호출 상한(설정값). 초과 시 호출 멈추고 경고.
- **기존 운영 안전망 계승**: recvWindow 15s·SL검증가드·시계동기화·autostart 자동복구는 exchange 계층 공유라 그대로 적용됨.

## 6. 테스트 전략

- **guard**: 순수 함수 → 표 기반 단위 테스트 강하게(위험 결정 거부·클램프·소액보정 각 케이스). 최우선.
- **memory**: Record/Recall/Stats 단위 테스트. 회고 라벨링 정확성.
- **trigger**: 합성 가격 시퀀스로 발화 조건·쿨다운 검증.
- **council**: LLMClient를 모의(mock)로 주입해 결정 파싱·HOLD 폴백·JSON 강제 검증. 실제 LLM은 통합테스트에서만.
- **runner**: 모의 exchange + 모의 council로 한 사이클 E2E.
- **전 패키지 `-race` green** + 기존 테스트 회귀 0 (현 프로젝트 기준 계승).

## 7. 단계적 구현 로드맵

1. **Phase 1 — 토대**: guard(가드) + memory(기억) 골격 + LLMClient 추상화. 페이퍼 전용. AI 없이 가드·기억 단위검증.
2. **Phase 2 — 두뇌**: council(Bull/Bear/Risk) + runner 엮기. 모의 LLM으로 파이프라인 검증 → 실제 모델 1종 연결.
3. **Phase 3 — 트리거·회고**: 이벤트 트리거 + 결과 회고 학습 가동. 페이퍼로 실시장 관찰.
4. **Phase 4 — 성과추적·전환제안**: 통계·전환 제안 알림. 페이퍼 성과 누적.
5. **Phase 5 — 실거래 전환**: 기준 충족 시 사용자 승인 하에 소액 실거래. 이후 증액도 동일.

각 Phase는 별도 plan→구현→검증 사이클. **Phase 1만 이번 plan 대상** (나머지는 진행하며 구체화).

## 9. 미확정 (구현 중 결정)

- 사용할 LLM 모델 (키 발급 시점에 사용자 확정 — 추상화돼 있어 영향 국소적)

- 전환 기준 구체 수치 (Phase 4에서 페이퍼 데이터 보고 튜닝)
